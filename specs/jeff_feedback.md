
The underlying EP3 API provides snapshot+streaming for many use cases. Where such a use case exists, any polling-based APIs should not be used by anyone external. Or they should be severely rate limited such that they are not useful for anything more than an occasional query. They put undue strain on the database. E.g. The underlying get_trade_stats (GetTradeStats), and similar. We do not have a data analytics database setup for external access that is optimized for high query load and our primary database getting overloaded puts risk on our lag from system -> database for the persistent writes. There are future infra solutions to help this but currently it is a limitation as we have not had the time to properly do the database optimization and setup for these high-load queries. We should either (a) not provide the documentation for the problematic APIs or (b) put a clear warning on them that low rate limits may be imposed at any time and tell people the correct use of the APIs is to snapshot/stream
